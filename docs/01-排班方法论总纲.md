# Gelato 门店排班：核心理念与逻辑框架

## 引言

> 排班不是填表格，而是**用有限的人力，在正确的时间、以足够的数量，覆盖门店的营业需求**。
>
> 这套框架回答三个问题：
> 1. **目标是什么？** — 排班要达成什么？
> 2. **从哪里开始？** — 面对一家完全不了解的门店，第一步做什么？
> 3. **为什么这么排？** — 背后的思考逻辑是什么？

---

## 一、排班的目标体系

排班不是单目标优化，而是**三个目标的平衡**：

```
     ┌──────────────┐
     │   服务覆盖    │
     │ (顾客不等待)  │
     └──────┬───────┘
            │
    ┌───────┴────────┐
    │    排班方案     │
    └───────┬────────┘
            │
┌───────────┴───────────┐
│ 人力成本     │ 员工可持续  │
│ (不超预算)   │ (休息充分)  │
└───────────────────────┘
```

### 目标 1：服务覆盖（第一优先级）
- 门店开门期间，每个时段都有足够员工
- 高峰期不因人力不足流失顾客
- 关键节点（开店、打烊）有人负责

### 目标 2：人力成本可控
- 总人时与营业额匹配（不养闲人）
- 加班费用在预算内

### 目标 3：员工可持续性
- 每周有足够休息
- 排班可预期、公平

> **三个目标有冲突时，优先级从上到下：服务覆盖 > 成本控制 > 员工偏好**
> 但在长期运营中，忽视员工可持续性最终会反过来伤害服务覆盖。

## 二、面对陌生门店：从零开始的诊断流程

> 你走进一家完全不了解的 Gelato 店，从哪里开始？

### 前提假设

为简化起点，我们做两个关键假设：
1. **全员通岗**：所有员工都能胜任所有岗位（收银、制作、清洁等）
2. **排班的核心问题简化为**：**每个时段需要几个人？**

> 实际运营中可能会有技能差异，但排班框架的第一步应该是"通岗假设"。
> 如果后期发现某些员工确实不能做某些事，再在框架上做减法。

### 第 0 步：先别排班，先观察

不要急着坐下排班。先去**看**和**问**：

#### 看什么（观察）

| 观察项 | 具体看什么 | 为什么重要 |
|--------|-----------|-----------|
| **客流节奏** | 一周七天，每天不同时段的客流变化 | 决定高峰/低谷划分 |
| **顾客等待时间** | 高峰期顾客排队多久 | 判断人力是否不足 |
| **营业高峰差异** | 周末和平时高峰的时段和强度是否不同 | 决定周末是否需要更多人 |
| **员工闲忙状态** | 忙的时候在做什么？闲的时候呢？ | 了解工作量分布 |

#### 问什么（访谈）

| 问题 | 问谁 | 目的 |
|------|------|------|
| "一周哪天最忙？哪天最闲？" | 店长 | 判断周间波动 |
| "周末和节假日客流跟平时比差多少？" | 店长 | 量化周末高峰增幅 |
| "高峰期每小时大概多少个顾客？" | 店长 | 为产能计算提供输入 |
| "一个人做一份 Gelato 大概多久？" | 员工 | 计算单人产能 |
| "现有排班有什么头疼的地方？" | 店长 | 了解痛点和改进方向 |

### 第 1 步：建立基础数据卡

```
┌─────────────────────────────────────────────┐
│  门店基础数据卡                             │
├─────────────────────────────────────────────┤
│  营业时间：10:00 - 22:00（12h）              │
│  员工人数：3 人（全部通岗）                  │
│  高峰时段（平时）：午 12-14 / 晚 17-19       │
│  高峰时段（周末）：午 11:30-14:30 / 晚 16-20 │
│  周末客流系数：平时的 1.5 倍                 │
│  单人产能：约 XX 单/小时                     │
│  高峰期预估客流量：XX 单/小时                │
│  现有痛点：[填写]                            │
└─────────────────────────────────────────────┘
```

### 第 2 步：诊断 → 找出核心矛盾

| 问题类型 | 典型症状 | 处方方向 |
|---------|---------|---------|
| **覆盖不足** | 高峰期排队太长、顾客投诉 | 增加高峰时段在岗人数 |
| **人力过剩** | 员工闲的时候太多、人时成本高 | 压缩班次或减少重叠 |
| **高峰期错配** | 平时够、周末不够 | 调整休息日，周末全员在岗 |
| **员工疲劳** | 离职率高、抱怨多 | 增加轮换、保证休息 |
| **缺乏流程** | 每次排班都很随意、无章可循 | 建立标准化排班流程 |

## 三、排班逻辑框架：为什么这么排？

### 核心原理：人员产能匹配客流需求

排班的本质是用**人的产出能力**去匹配**顾客的需求量**：

```
预估客流（单/小时） ÷ 单人产能（单/小时） = 最低在岗人数
```

### 第一条逻辑线：客流预测 → 知道"忙不忙"

#### 三个层级的客流分析

```
第一层：日高峰
        一天内什么时候人多？
        回答 → 午高峰（12-14）和晚高峰（17-19）

第二层：周高峰（新增）
        一周内哪天人多？
        回答 → 周末客流是工作日的 1.5 倍
               节假日客流可达工作日的 2 倍
               
第三层：季节性高峰
        一年内哪几个月人多？
        回答 → 夏季 Gelato 旺季 / 冬季淡季
```

**休息日安排的逻辑**：
```
高峰日（周末/节假日）→ 尽量全员在岗
低峰日（周中）      → 安排轮流休息
```

#### 示例：3 人店一周高峰分布

```
        周一  周二  周三  周四  周五  周六  周日
客流水平 低    低    中    中    高    极高   高
休息安排 可休  可休   —     —     —   不可休   —
在岗人数  2    2     3     3     3     3     3
```

### 第二条逻辑线：产能计算 → 知道"需要几个人"

#### 核心公式

```
最低在岗人数 = ⌈ 高峰每小时客流量（单） ÷ 单人每小时产能（单） ⌉

例子：
  高峰每小时预估 = 60 单
  每人每小时能做 = 20 单
  最低在岗人数 = 60 ÷ 20 = 3 人 ✅
```

#### 单人产能如何确定

| 方法 | 说明 | 适用场景 |
|------|------|---------|
| **直接计算** | 1 份 Gelato 制作时间 + 收银时间 | 有出单数据 |
| **观察估算** | 观察高峰 1 小时，记录总出单数 ÷ 当时人数 | 无精确数据 |
| **行业参考** | Gelato 店平均单人产值 | 完全不了解时 |

> **关键原则：产能估算是会不断修正的。**
> 第一周先用估算值，跑通后再根据实际数据调整。

#### 示例：3 人 Gelato 店

```
场景：周五晚高峰（18:00-19:00）
    预估客流量 = 50 单
    单人产能 = 18 单/小时（每单约 3.3 分钟）
    计算结果 = 50 ÷ 18 ≈ 2.8 → 需要 3 人

场景：周一上午（10:00-12:00）
    预估客流量 = 15 单
    单人产能 = 18 单/小时
    计算结果 = 15 ÷ 18 ≈ 0.8 → 需要 1-2 人
```

### 第三条逻辑线：营业时间 → 班次结构

```
营业时间 ≤ 8h → 一个班次就够了
营业时间 8-12h → 需要 2-3 个班次错峰
营业时间 12h+ → 必须三班倒或加兼职
```

**为什么？** 每人每天最多工作 8 小时。营业时间超出 8 小时的部分，必须用第二个人的时间来覆盖。

### 三线合成 → 决策树

```
面对一家新店：
│
├─ 问：营业多久？
│  ├─ ≤8h → 一个班次即可
│  └─ >8h → 需要多班次错峰
│
├─ 问：几个员工？
│  ├─ 1人 → 只能缩短营业时间
│  ├─ 2人 → 错班，但人手很紧
│  ├─ 3人 → 三班错峰 ✅
│  └─ 4人+ → 固定班次+弹性
│
├─ 问：周末和平时客流差异大吗？
│  ├─ 不大 → 每天同样排法
│  └─ 很大 → 分平日版/周末版排班
│
├─ 问：单人产能多少？高峰客流多少？
│  ├─ 算出最低在岗人数
│  └─ 跟可用人数比较，决定是否需要调整
│
└─ 问：现有排班有什么问题？
   ├─ 排队时间长 → 产能不足 → 加人或优化流程
   ├─ 成本太高 → 减重叠或调休息日
   └─ 员工抱怨 → 加轮换
```

## 四、本店示例：框架如何落地

### 门店基础数据

| 项目 | 数据 |
|------|------|
| 营业时间 | 12h（10:00-22:00） |
| 员工 | 3 人，全部通岗 |
| 单人产能 | ~18 单/小时（估算） |
| 平时高峰 | 午 12-14 / 晚 17-19，45-50 单/小时 |
| 周末高峰 | 午 11:30-14:30 / 晚 16-20，60-70 单/小时 |

### 产能计算 → 在岗人数

| 时段 | 预估客流 | 单人产能 | 计算需求 | 实际安排 |
|------|---------|---------|---------|---------|
| 平时高峰 | 50 单/h | 18 单/h | 2.8 → 3 人 | A+B+C 全员 |
| 周末高峰 | 65 单/h | 18 单/h | 3.6 → 4 人 | 仅 3 人，存在缺口 |
| 平时平峰 | 20 单/h | 18 单/h | 1.1 → 2 人 | 2 人在岗 |
| 低峰时段 | 10 单/h | 18 单/h | 0.6 → 1 人 | 1-2 人 |

> ⚠️ 周末高峰 65 单/h 时，3 人最多出 54 单（18×3），缺口 11 单
> → 对策：周末高峰期可以提前备料，简化流程，或考虑招兼职

### 由框架推导出的班次方案

| 框架问题 | 回答 | 推导结论 |
|---------|------|---------|
| 营业多久？ | 12h | → 多班次错峰 |
| 几个员工？ | 3 人通岗 | → 三班制 |
| 产能算出来需要几人？ | 高峰 3 人 | → 全员在岗 |
| 周末和平时差异大吗？ | 周末多 30%+ | → 周末不排休，平时轮休 |

**最终方案：A(10-18)/B(12-20)/C(14-22) 三班，周末全员，工作日轮休**

## 五、持续优化：如何越排越好

### 迭代闭环

```
试排一周 → 对比实际客流与预估 → 修正产能参数 → 调整排班 → 再试
```

### 每周复盘

每周末花 10 分钟回答：

| 问题 | 怎么判断 |
|------|---------|
| 高峰期排队了吗？排队多久？ | 观察记录 |
| 预估客流和实际差多少？ | 对比数据 |
| 产能估算准不准？ | 用实际单数重新算 |
| 周末安排合理吗？ | 员工反馈 + 客流数据 |
| 休息日安排满意吗？ | 员工反馈 |

### 参数修正

每次复盘后，更新基础数据卡中的参数：

- 单人产能（单/小时）→ 根据实际出单数据校准
- 高峰时段客流 → 根据实际观察调整
- 周末/节假日系数 → 根据数据修正

## 六、框架 vs 手册：两者的关系

```
┌──────────────────────────────────────────────┐
│  【框架层】核心理念与逻辑                    │
│                                              │
│  · 目标体系（什么重要）                       │
│  · 诊断方法（从哪里开始）                     │
│  · 产能计算（需要几个人）                     │
│  · 周内高峰（休息日怎么排）                  │
│  · 决策树（不同情况怎么选）                   │
├──────────────────────────────────────────────┤
│  【执行层】操作手册 + 表格模板                │
│                                              │
│  · Step-by-step 排班步骤                      │
│  · 具体操作指引                               │
│  · 表格模板公式                               │
│  · 检查清单                                   │
└──────────────────────────────────────────────┘
```

**框架层回答"为什么"和"从哪开始"**
**执行层回答"怎么做"**

两者结合，才能让使用者不仅会排班，还**理解排班**——面对任何一家新店，都能独立完成从诊断到方案的全过程。
